== Controller Service

The `JwtIssuerConfigService` manages issuer configuration and token validation settings. Both processors reference the same CS instance.

=== Controller Service Properties

[cols="1,1,3,1"]
|===
|Property |Default |Description |Required

|jwt.validation.jwks.refresh.interval
|3600
|Interval in seconds for refreshing JWKS keys
|Yes

|jwt.validation.jwks.connection.timeout
|10
|Seconds before a JWKS endpoint request times out
|Yes

|jwt.validation.maximum.token.size
|16384
|Maximum token size in bytes
|Yes

|jwt.validation.allowed.algorithms
|Secure defaults
|Comma-separated list of permitted signing algorithms (RS256, RS384, RS512, ES256, ES384, ES512, PS256, PS384, PS512). The `none` algorithm is never allowed regardless of this setting.
|No

|jwt.validation.require.https.for.jwks
|true
|Enforces HTTPS for JWKS endpoint URLs. Disable only for development.
|Yes

|jwt.validation.jwks.allow.private.network.addresses
|false
|Whether to allow JWKS URLs pointing to private network addresses. Enable only for development.
|Yes
|===

=== Dynamic Properties (Issuers)

The Controller Service supports multiple issuers through NiFi dynamic properties. Issuer properties use the prefix `issuer.<name>.<property>` (where `<name>` is any identifier string, e.g., `keycloak`, `auth0`, or numeric like `1`, `2`):

[cols="1,3,1"]
|===
|Property |Description |Example

|issuer.<name>.jwks-url
|JWKS endpoint URL (for file-based JWKS, use `issuer.<name>.jwks-file` instead)
|https://auth.example.com/.well-known/jwks.json

|issuer.<name>.enabled
|Enable or disable this issuer without removing the configuration
|true

|issuer.<name>.audience
|Required `aud` claim for tokens from this issuer
|api://my-service

|issuer.<name>.name
|Issuer identifier name (alternative to `issuer.<name>.issuer`); at least one of `name` or `issuer` is required. *Warning*: This value becomes the `issuerIdentifier` used for token matching. If both `name` and `issuer` are set, `name` takes precedence. Use a URI matching the `iss` claim.
|https://auth.example.com/

|issuer.<name>.issuer
|Issuer URI (must match the `iss` claim in the JWT)
|https://auth.example.com/

|issuer.<name>.jwks-file
|File path to a local JWKS key file (alternative to `jwks-url`)
|/opt/nifi/conf/jwks.json

|issuer.<name>.jwks-content
|Inline JWKS JSON content. *Not yet supported*, the backend logs a warning and skips issuers configured with this option. Use `jwks-url` or `jwks-file` instead.
|`{"keys":[...]}`

|issuer.<name>.jwks-type
|Explicit JWKS source type: `url` or `file` (auto-detected if omitted). The `content` type is recognized but not yet supported.
|url

|issuer.<name>.client-id
|Expected client ID claim for tokens from this issuer
|my-client-app

|===

NOTE: Roles and scopes are configured at the **processor level** (via `Required Roles` / `Required Scopes` on the `MultiIssuerJWTTokenAuthenticator`, or via `restapi.<name>.required-roles` / `restapi.<name>.required-scopes` on the `RestApiGatewayProcessor`), not as per-issuer dynamic properties.

NOTE: In static configuration files, the prefix is `jwt.validation.issuer.<name>.<property>` (see <<Static Configuration Files>>).

== Route Configuration via Dynamic Properties

Routes are configured using NiFi dynamic properties with the `restapi.` prefix:

[cols="1,3,1"]
|===
|Property Pattern |Description |Example

|restapi.<name>.path |URL path for the route (exact match) |/api/users
|restapi.<name>.methods |Comma-separated HTTP methods (default: GET,POST,PUT,DELETE) |GET,POST
|restapi.<name>.required-roles |Comma-separated roles the JWT must carry |admin,editor
|restapi.<name>.required-scopes |Comma-separated scopes the JWT must carry |read,write
|restapi.<name>.success-outcome |NiFi relationship name; auto-populated with route name when absent |api-data
|restapi.<name>.create-flowfile |Whether to create a FlowFile (default: true) |false
|restapi.<name>.auth-mode |Authentication mode: `bearer` (default), `none`, `local-only` |none
|restapi.<name>.max-request-size |Per-route max body size in bytes (0 = global default) |524288
|restapi.<name>.enabled |Whether the route is active (default: true) |false
|restapi.<name>.schema |File path to JSON Schema for request body validation |./conf/schemas/user.json
|restapi.<name>.tracking-mode |Tracking mode: `none` (default), `simple`, `attachments` (see link:attachments-api.adoc[Attachments API]) |simple
|restapi.<name>.attachments-min-count |Minimum number of attachments required before auto-transition to `PROCESSED` (default: 0). Only relevant when `tracking-mode=attachments`. |2
|restapi.<name>.attachments-max-count |Maximum number of attachments allowed; 0 = use global hard limit. Only relevant when `tracking-mode=attachments`. |5
|restapi.<name>.attachments-timeout |Time window for attachment uploads (default: `30 sec`). Only relevant when `tracking-mode=attachments`. |60 sec
|===

=== Multi-Route Configuration Example

[source,properties]
----
# Route 1: User API (read-only)
restapi.users.path = /api/users
restapi.users.methods = GET
restapi.users.required-roles = read

# Route 2: Admin API (full access)
restapi.admin.path = /api/admin
restapi.admin.methods = GET,POST,PUT,DELETE
restapi.admin.required-roles = admin
restapi.admin.required-scopes = admin:write

# Route 3: Public health check (no authentication)
restapi.public.path = /api/public
restapi.public.methods = GET
restapi.public.auth-mode = none
restapi.public.create-flowfile = false
----

== Static Configuration Files

For container or automated deployments, the Controller Service loads configuration from `$NIFI_HOME/conf/cui-nifi-extensions.properties` or `$NIFI_HOME/conf/cui-nifi-extensions.yml`. The properties file is checked first. NiFi UI settings take precedence over static configuration, allowing runtime overrides of file-based defaults.

image::configuration-properties.png[Configuration Properties Hierarchy, align="center"]

=== Properties Format

[source,properties]
----
jwt.validation.issuer.1.issuer=https://auth.example.org/
jwt.validation.issuer.1.enabled=true
jwt.validation.issuer.1.jwks-url=https://auth.example.org/.well-known/jwks.json
jwt.validation.issuer.1.audience=my-api

jwt.validation.issuer.2.issuer=https://auth2.example.org/
jwt.validation.issuer.2.enabled=false
jwt.validation.issuer.2.jwks-url=https://auth2.example.org/.well-known/jwks.json
jwt.validation.issuer.2.audience=my-secondary-api
----

=== YAML Format

NOTE: The YAML format uses `issuers` (plural, list syntax) while the properties format uses `issuer` (singular, flat key syntax). The `ConfigurationManager` handles both formats transparently.

[source,yaml]
----
jwt:
  validation:
    issuers:
      - issuer: "https://auth.example.org/"
        enabled: true
        jwks-url: "https://auth.example.org/.well-known/jwks.json"
        audience: "my-api"
      - issuer: "https://auth2.example.org/"
        enabled: false
        jwks-url: "https://auth2.example.org/.well-known/jwks.json"
        audience: "my-secondary-api"
----

=== Reload Behavior

Configuration files are loaded once when the Controller Service is enabled (`@OnEnabled`). The `ConfigurationManager` includes a `checkAndReloadConfiguration()` method that can detect file changes, but it is not currently invoked automatically by either processor. To apply configuration file changes, disable and re-enable the Controller Service.

== Environment Variables

[cols="2,1,3"]
|===
|Variable |Type |Description

|`JWT_JWKS_REFRESH_INTERVAL` |Integer |JWKS cache refresh interval in seconds (e.g. `3600`). Stored as `jwt.validation.jwks.refresh.interval` in the Controller Service's static properties.
|`JWT_ISSUER_{NAME}_JWKS_URL` |URL |JWKS endpoint for an issuer. Note: environment variable underscores are converted to dots internally (`JWKS_URL` becomes `jwks.url`), which does not match the expected property key `jwks-url`. Use static configuration files instead for issuer JWKS URLs.
|===

== Custom UI REST Endpoints

The Custom UI backend provides REST endpoints shared by both processors:

=== Token Verification

* *Path*: `/nifi-api/processors/jwt/verify-token`
* *Method*: POST
* Verifies JWT tokens using the same configuration as the processor
* Isolated, does not affect processor metrics

=== JWKS Validation

* *Path*: `/nifi-api/processors/jwt/validate-jwks-url` (and `-file`, `-content`)
* *Method*: POST
* Stateless JWKS structure and accessibility validation

=== Gateway Proxy Endpoints

[cols="1,1,2"]
|===
|UI Endpoint |Method |Description

|`/nifi-api/processors/jwt/gateway/config` |GET |Retrieves current route configuration
|`/nifi-api/processors/jwt/gateway/metrics` |GET |Retrieves metrics from the gateway `/metrics` endpoint
|`/nifi-api/processors/jwt/gateway/test` |POST |Sends test requests to the gateway (SSRF-protected)
|`/nifi-api/processors/jwt/gateway/token-fetch` |POST |Proxies an OAuth token request to the IDP
|`/nifi-api/processors/jwt/gateway/discover-token-endpoint` |POST |Fetches OIDC discovery for the token endpoint
|`/nifi-api/processors/jwt/gateway/health` |GET |Proxies to gateway `/health` endpoint
|===
