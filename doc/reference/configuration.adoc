=== Dynamic Properties (Issuers)

The Controller Service supports multiple issuers through NiFi dynamic properties. Issuer properties use the prefix `issuer.<name>.<property>` (where `<name>` is any identifier string, e.g., `keycloak`, `auth0`, or numeric like `1`, `2`):

[cols="1,3,1"]
|===
|Property |Description |Example

|issuer.<name>.jwks-url
|JWKS endpoint URL (for file-based JWKS, use `issuer.<name>.jwks-file` instead)
|https://auth.example.com/.well-known/jwks.json

|issuer.<name>.enabled
|Enable or disable this issuer without removing the configuration
|true

|issuer.<name>.audience
|Required `aud` claim for tokens from this issuer
|api://my-service

|issuer.<name>.name
|Issuer identifier name (alternative to `issuer.<name>.issuer`); at least one of `name` or `issuer` is required. *Warning*: This value becomes the `issuerIdentifier` used for token matching. If both `name` and `issuer` are set, `name` takes precedence. Use a URI matching the `iss` claim.
|https://auth.example.com/

|issuer.<name>.issuer
|Issuer URI (must match the `iss` claim in the JWT)
|https://auth.example.com/

|issuer.<name>.jwks-file
|File path to a local JWKS key file (alternative to `jwks-url`)
|/opt/nifi/conf/jwks.json

|issuer.<name>.jwks-content
|Inline JWKS JSON content. *Not yet supported*, the backend logs a warning and skips issuers configured with this option. Use `jwks-url` or `jwks-file` instead.
|`{"keys":[...]}`

|issuer.<name>.jwks-type
|Explicit JWKS source type: `url` or `file` (auto-detected if omitted). The `content` type is recognized but not yet supported.
|url

|issuer.<name>.client-id
|Expected client ID claim for tokens from this issuer
|my-client-app

|===

NOTE: Roles and scopes are configured at the **processor level** (via `Required Roles` / `Required Scopes` on the `MultiIssuerJWTTokenAuthenticator`, or via `restapi.<name>.required-roles` / `restapi.<name>.required-scopes` on the `RestApiGatewayProcessor`), not as per-issuer dynamic properties.

NOTE: In static configuration files, the prefix is `jwt.validation.issuer.<name>.<property>` (see <<Static Configuration Files>>).

[#reverse-proxy-forwarded-header-resolution]
=== Reverse-Proxy / Forwarded Header Resolution

When the gateway sits behind a reverse proxy (nginx, HAProxy, an ingress
controller), the proxy advertises the original request context via the
`X-Forwarded-*` / `Forwarded` header family. All of this resolution is delegated to
the shared `ForwardedRequestResolver` (a thin wrapper over the cui-http
`de.cuioss.http.forwarded.ForwardedHeaderResolver`), so the trust model, header
precedence, sanitization, and injection guards live in exactly one place and the UI
servlet, the REST gateway, and the integration tests resolve identically.

The resolver honors four independent forwarded aspects, each *secure by default*
(honored only under explicit opt-in):

* *Context-path prefix* (`X-ProxyContextPath` / `X-Forwarded-Prefix`) — gated by the
allowlist or trust-all (below).
* *Scheme / host / port* (`X-Forwarded-Proto` / `-Host` / `-Port`) — reflected in the
absolute 202 `Location` header; honored only under trust-all.
* *Client IP* (`X-Forwarded-For` / `Forwarded`) — used for audit and rate-limit
logging; honored only when `rest.gateway.proxy.trusted-proxies` is configured.
* *Sanitization posture* (`rest.gateway.proxy.security-config.preset`) — governs how
strictly the header values are sanitized before any of the above are honored.

==== Context Path (Trusted-Proxy Allowlist)

The proxy advertises its path prefix via the `X-ProxyContextPath` header
(NiFi-native) or the de-facto `X-Forwarded-Prefix` header. The gateway uses the
honored prefix for two things:

* *Inbound route matching* — the prefix is stripped before routing, so a proxied
request such as `GET /nifi-proxy/health` resolves to the `/health` handler.
* *Outbound URL generation* — the prefix is prepended to the 202 `Location` header
and to the HATEOAS `_links` hrefs, so the URLs the client receives point back
through the proxy.

Because the embedded Jetty server can listen on `0.0.0.0`, a direct client could
otherwise *spoof* these headers to manipulate routing and the reflected prefix.
The gateway therefore honors a proxy context path *only when it appears in the
operator-configured allowlist* `rest.gateway.proxy.context-path.whitelist`
(comma-separated). This mirrors NiFi's own `nifi.web.proxy.context.path` allowlist
model.

* *Default (empty allowlist)* — the gateway *ignores* client-supplied proxy
context-path headers entirely and does not trust them. Routing and generated URLs
use the raw request path. This is secure by default: a direct-exposure deployment
cannot be spoofed.
* *Behind a reverse proxy* — list the prefix(es) the proxy sets, e.g.
`rest.gateway.proxy.context-path.whitelist = /nifi-proxy`. Only listed prefixes are
honored; an unlisted (spoofed) prefix is ignored and the request is matched against
its raw path (typically yielding a 404).

The allowlist gates *both* inbound route matching *and* the outbound
`Location` / `_links` prefix, so the two always agree.

==== Trust-All Override

The `rest.gateway.proxy.context-path.trust-all` toggle (boolean, default `false`)
sits on top of the allowlist. When set to `true`, the gateway honors *any*
resolved-and-normalized proxy context path from the headers *without* checking the
allowlist — the header value is still normalized and injection-guarded (CR/LF,
protocol-relative `//host`, and backslash values are rejected), but the allowlist
membership check is bypassed entirely.

[WARNING]
====
Enable trust-all *only* when the gateway is exclusively reachable through a trusted
reverse proxy that sets these headers. A directly-reachable listener (for example
bound to `0.0.0.0`) with trust-all enabled lets *any* client spoof the context path,
manipulating routing and the reflected prefix in generated URLs. When in doubt, leave
trust-all `false` and use the allowlist.
====

* *`trust-all = false` (default)* — the allowlist in
`rest.gateway.proxy.context-path.whitelist` governs which context paths are honored,
as described above.
* *`trust-all = true`* — every normalized prefix is honored and the allowlist is
ignored.

When a proxy context-path header arrives but is *not* honored specifically because
the allowlist is empty *and* trust-all is `false` — the "positive-list active but
empty" misconfiguration, i.e. the gateway is behind a proxy but has not been
configured to honor it — the gateway logs a one-shot `WARN` (per handler instance)
naming *both* config keys so the operator can fix it. No warning is logged when
trust-all is `true`, when no proxy header is present, or when a *non-empty* allowlist
simply does not contain the received prefix (that is a deliberate reject, not a
misconfiguration).

==== Forwarded Scheme / Host / Port (Location header)

When the gateway accepts a tracked request it returns a 202 `Accepted` with an
absolute `Location` header pointing at the request-status endpoint. Each of the
scheme, host, and port prefers the honored forwarded value
(`X-Forwarded-Proto` / `X-Forwarded-Host` / `X-Forwarded-Port`), falling back
per-field to the raw Jetty request value when that field was not honored — so a
reverse-proxy-facing URL is reflected under opt-in and the URI is byte-identical to
the raw request otherwise. The honored context-path prefix (above) is prepended.

The forwarded scheme/host/port are honored *only when
`rest.gateway.proxy.context-path.trust-all` is `true`* (the same trust gate as the
context path). Under the secure default (`trust-all = false`) the `Location` header
uses the raw request scheme/host/port, exactly as before.

==== Forwarded Client IP (audit / rate-limit logging)

The gateway's audit log entries and rate-limit keys use the *client IP*. By default
this is the raw socket remote address. When
`rest.gateway.proxy.trusted-proxies` is configured (a comma-separated list of IP
literals or CIDR ranges), the gateway instead resolves the client IP from the
`X-Forwarded-For` / `Forwarded` chain: the chain is walked *right-to-left*, skipping
every hop that matches a trusted-proxy entry, and the first *untrusted* address is
used as the client IP.

* *Default (empty)* — no forwarded client IP is honored; the raw socket remote
address is used. This is secure by default: a direct client cannot forge the
`X-Forwarded-For` chain.
* *Configured* — list the trusted proxy hops, e.g.
`rest.gateway.proxy.trusted-proxies = 10.0.0.0/8, 192.168.1.5`. Enable this *only*
when the gateway is exclusively reachable through the listed proxies — a
directly-reachable listener lets any client forge the chain.

==== Sanitization Preset (security-config)

`rest.gateway.proxy.security-config.preset` selects the security posture of the
forwarded-header sanitization pipeline that runs *before* any forwarded value is
honored:

* `strict` — tightest restrictions.
* `lenient` — maximum compatibility; trusted environments only.
* `defaults` (the default) — balanced.

The preset governs how the `X-Forwarded-*` / `Forwarded` header values are sanitized
(control-character, CR/LF-injection, protocol-relative, and backslash rejection)
regardless of which forwarded aspects are opted into above.

==== UI Context-Path Servlet

The custom UI resolves the reverse-proxy context path through the same shared
`ForwardedRequestResolver`. The UI `ProxyContextPathServlet` derives its honored
allowlist from NiFi's own `nifi.web.proxy.context.path` property (parsed the same way
as NiFi's native proxy-context-path allowlist), so an existing proxied UI deployment
that already sets that property keeps working with no extra configuration. An optional
`allowedContextPaths` (and `trustAll`) servlet init-param in the UI `web.xml` overrides
the NiFi-derived value when present. With neither the property nor the init-param set
the allowlist is empty and the servlet honors nothing — every prefix resolves to the
empty string (secure default).

[#path-parameter-routes]
=== Path-Parameter Routes

A route `path` may contain `{placeholder}` segments to capture variable parts of the request URI. The placeholder values are extracted from the matched request and written to the FlowFile as `rest.api.pathparam.<name>` attributes (one per placeholder).

Placeholder syntax:

* `{name}` -- matches a single path segment.
* `{name:regex}` -- matches a single segment constrained by `regex`, e.g. `{id:\\d+}`.

Routes resolve through three ordered passes -- exact match, prefix match, then `{placeholder}` pattern match -- and the first to hit wins. A literal route (for example `/api/users/me`) therefore takes precedence over a pattern route (`/api/users/{id}`) that would also match.

[source,properties]
----
# Path-parameter route: /api/users/42/orders/7 → userId=42, orderId=7
restapi.user-orders.path = /api/users/{userId}/orders/{orderId}
restapi.user-orders.methods = GET
restapi.user-orders.required-roles = read
----

For a `GET /api/users/42/orders/7` request, the resulting FlowFile carries:

[cols="2,1"]
|===
|Attribute |Value

|`rest.route.name` |`user-orders`
|`http.request.uri` |`/api/users/42/orders/7`
|`rest.api.pathparam.userId` |`42`
|`rest.api.pathparam.orderId` |`7`
|===

See link:../architecture/gateway.adoc#path-parameter-routes[Gateway Architecture -- Path-Parameter Routes] for the resolution pipeline and match precedence.

== Static Configuration Files

For container or automated deployments, both the Controller Service and the `RestApiGatewayProcessor` load configuration from `$NIFI_HOME/conf/cui-nifi-extensions.properties` or `$NIFI_HOME/conf/cui-nifi-extensions.yml`. The properties file is checked first. NiFi UI settings take precedence over static configuration, allowing runtime overrides of file-based defaults. The Controller Service reads issuer definitions (`jwt.validation.issuer.N.*`) from these files; the `RestApiGatewayProcessor` reads its `restapi.*` route definitions from the same files.

NOTE: The path is resolved relative to the NiFi JVM's working directory, which is `$NIFI_HOME` in a standard container launch but may differ if NiFi is started from another directory. Only the issuer (`jwt.validation.issuer.N.*`) and `restapi.*` keys are consumed from static files. Non-issuer `jwt.validation.*` settings (JWKS refresh interval, connection timeout, token size, allowed algorithms, HTTPS enforcement) are **not** read from static files or environment variables — those values come exclusively from the Controller Service's NiFi property descriptors, which always carry defaults, so a static-file or environment value for them can never take effect.

=== YAML Format

NOTE: The YAML format uses `issuers` (plural, list syntax) while the properties format uses `issuer` (singular, flat key syntax). The `ConfigurationManager` handles both formats transparently.

[source,yaml]
----
jwt:
  validation:
    issuers:
      - issuer: "https://auth.example.org/"
        enabled: true
        jwks-url: "https://auth.example.org/.well-known/jwks.json"
        audience: "my-api"
      - issuer: "https://auth2.example.org/"
        enabled: false
        jwks-url: "https://auth2.example.org/.well-known/jwks.json"
        audience: "my-secondary-api"
----

=== Reload Behavior

Configuration files are loaded once when the Controller Service is enabled (`@OnEnabled`). The `ConfigurationManager` includes a `checkAndReloadConfiguration()` method that can detect file changes, but it is not currently invoked automatically by either processor. To apply configuration file changes, disable and re-enable the Controller Service.

== Custom UI REST Endpoints

The Custom UI backend provides REST endpoints shared by both processors:

=== Token Verification

* *Path*: `/nifi-api/processors/jwt/verify-token`
* *Method*: POST
* Verifies JWT tokens using the same configuration as the processor
* Isolated, does not affect processor metrics

=== JWKS Validation

* *Path*: `/nifi-api/processors/jwt/validate-jwks-url` (and `-file`, `-content`)
* *Method*: POST
* Stateless JWKS structure and accessibility validation

=== Gateway Proxy Endpoints

[cols="1,1,2"]
|===
|UI Endpoint |Method |Description

|`/nifi-api/processors/jwt/gateway/config` |GET |Retrieves current route configuration
|`/nifi-api/processors/jwt/gateway/metrics` |GET |Retrieves metrics from the gateway `/metrics` endpoint (see link:metrics-api.adoc[Metrics API])
|`/nifi-api/processors/jwt/gateway/test` |POST |Sends test requests to the gateway (SSRF-protected)
|`/nifi-api/processors/jwt/gateway/token-fetch` |POST |Proxies an OAuth token request to the IDP
|`/nifi-api/processors/jwt/gateway/discover-token-endpoint` |POST |Fetches OIDC discovery for the token endpoint
|`/nifi-api/processors/jwt/gateway/health` |GET |Proxies to gateway `/health` endpoint
|===
