= Attachment Merge Integration Guide
:toc:
:toc-placement: preamble
:imagesdir: ../plantuml

Practical guide for integrating the attachment merge pattern into NiFi flows using the REST API Gateway processor.

== Overview

When a route uses `tracking-mode=attachments`, the REST API Gateway accepts attachments via `POST /attachments/{parentTraceId}` and automatically transitions the parent status to `PROCESSED` once the minimum attachment count is met. The Wait/Notify pattern in the NiFi flow holds the parent FlowFile until attachments arrive (or a timeout expires), then releases it for downstream processing. This guide covers the full lifecycle from initial upload through `PROCESSED` status, with curl examples against the sandbox environment.

== Quick Start: curl Walkthrough

These examples use the sandbox environment started via `run-and-deploy.sh`. Adjust the base URL if your deployment differs.

=== 1. Get a Token

[source,bash]
----
TOKEN=$(curl -sk -X POST \
  'https://localhost:9085/realms/oauth_integration_tests/protocol/openid-connect/token' \
  -d 'grant_type=password' \
  -d 'client_id=integration-test-client' \
  -d 'username=testuser' \
  -d 'password=testpassword' \
  | jq -r '.access_token')
----

=== 2. POST /api/upload

[source,bash]
----
curl -sk -X POST https://localhost:9443/api/upload \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"document": "example-payload"}' | jq .
----

Response (202):
[source,json]
----
{
  "status": "accepted",
  "traceId": "aaaaaaaa-bbbb-cccc-dddd-eeeeeeeeeeee",
  "_links": {
    "status": { "href": "/status/aaaaaaaa-bbbb-cccc-dddd-eeeeeeeeeeee" },
    "attachments": { "href": "/attachments/aaaaaaaa-bbbb-cccc-dddd-eeeeeeeeeeee" }
  }
}
----

=== 3. GET /status/{traceId}

[source,bash]
----
curl -sk https://localhost:9443/status/$TRACE_ID \
  -H "Authorization: Bearer $TOKEN" | jq .
----

Response: `"status": "COLLECTING_ATTACHMENTS"`

=== 4. POST /attachments/{traceId}

[source,bash]
----
curl -sk -X POST https://localhost:9443/attachments/$TRACE_ID \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"file": "attachment-content"}' | jq .
----

Response (202): accepted with its own `traceId` and a `_links.status` link.

=== 5. GET /status/{traceId} (after attachment submitted)

Check the parent status. Once the minimum attachment count (`attachments-min-count`) is met, the status transitions to `PROCESSED` synchronously:

[source,bash]
----
curl -sk https://localhost:9443/status/$TRACE_ID \
  -H "Authorization: Bearer $TOKEN" | jq .status
----

Response: `"PROCESSED"` (transitions immediately when min count is reached).

=== 6. POST /attachments/{traceId} (additional attachments)

After the status transitions to `PROCESSED`, additional attachments are still accepted up to `attachments-max-count`. Once the max count is reached:

[source,bash]
----
curl -sk -X POST https://localhost:9443/attachments/$TRACE_ID \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"file": "over-limit"}' | jq .
----

Response (409): `"Attachment limit reached: <max-count>"`.

NOTE: If the parent status has transitioned to `PROCESSING`, `REJECTED`, or `ERROR` (e.g., by downstream NiFi flow logic), the attachment window closes and further uploads return 409 `"Attachment window closed — parent request is already being processed"`.

== Status Lifecycle

image::attachment-status-lifecycle.png[Attachment Status Lifecycle]

|===
| Status | Meaning | Attachment Window

| `COLLECTING_ATTACHMENTS`
| Parent request accepted, waiting for attachments
| *Open*: uploads accepted

| `PROCESSED`
| Minimum attachment count met (auto-transitioned by gateway)
| *Open*: uploads still accepted up to max count

| `PROCESSING`
| Downstream flow processing (set by NiFi flow if needed)
| *Closed*: uploads rejected with 409
|===

== NiFi Flow Setup for Integrators

=== Prerequisites

* *DistributedMapCacheServer* controller service (e.g., port 4557)
* *DistributedMapCacheClientService* controller service pointing to the server
* *RestApiGateway* processor with a route configured for `tracking-mode=attachments`

=== Wait/Notify Pattern

The Wait processor holds the parent FlowFile until attachments arrive. The Notify processor (triggered by each attachment) increments the signal counter.

[cols="1,2,3"]
|===
| Processor | Property | Value

| Wait
| Signal ID
| `merge-signal-${rest.trace.id}`

| Wait
| Target Signal Count
| `${rest.trace.attachments.min.count:replaceEmpty('1')}`

| Wait
| Expiration Duration
| `${rest.trace.attachments.timeout:replaceEmpty('30 sec')}`

| Notify
| Signal ID
| `merge-signal-${rest.trace.parent.id}`

| Notify
| Counter Delta
| `1`
|===

The Notify processor uses `rest.trace.parent.id` (not `rest.trace.id`) because attachments reference their parent's signal. Each attachment FlowFile carries the parent trace ID in this attribute.

=== Status Transition (Java-based)

The `COLLECTING_ATTACHMENTS` → `PROCESSED` transition is handled automatically by the `AttachmentsEndpointHandler` in Java code. When the number of accepted attachments reaches `attachments-min-count`, the handler calls `RequestStatusStore.updateStatus()` to set the parent status to `PROCESSED`. This is synchronous and idempotent — no NiFi flow processors are needed for this transition.

The Wait/Notify pattern in the NiFi flow is used only for FlowFile synchronization (holding the parent FlowFile until attachments arrive), not for status updates.

==== Custom transitions (production)

For production flows that need additional status transitions after the Wait processor releases the parent FlowFile, use the standard NiFi approach of `UpdateAttribute` → `PutDistributedMapCache`:

* `PROCESSED` → `PROCESSING`: if downstream business logic needs an intermediate state
* `PROCESSING` → `REJECTED`: route validation failures
* `PROCESSING` → `ERROR`: route processing errors

=== Route Configuration Properties

[source,properties]
----
restapi.upload.tracking-mode=attachments
restapi.upload.attachments-min-count=1
restapi.upload.attachments-max-count=5
restapi.upload.attachments-timeout=30 sec
----

The timeout value is exposed as the `rest.trace.attachments.timeout` FlowFile attribute, which the Wait processor reads via Expression Language. The timeout field appears in the route editor UI alongside min/max attachment count fields (only visible when `tracking-mode` is set to `attachments`).

== Timeout and Edge Cases

=== Timeout (no attachments arrive)

When the Wait processor's expiration duration elapses without reaching the target signal count, the parent FlowFile is released via the `expired` relationship and routed to `LogAttribute (gateway-merged)`. The status remains at `COLLECTING_ATTACHMENTS` in the cache unless the min count was already met (in which case it is already `PROCESSED`).

=== Multiple attachments before min count

Each `POST /attachments/{parentTraceId}` triggers a Notify that increments the signal counter by 1. The Wait processor only releases the parent when the counter reaches `min-count`. Until then, the parent remains in the Wait loop.

=== Max count enforcement

The REST API Gateway enforces `attachments-max-count` at the HTTP level. Once the count is reached, additional `POST /attachments/{parentTraceId}` requests return *409 Conflict* with `application/problem+json` before any FlowFile is created.

=== Attachment after window closed

When the parent status is set to `PROCESSING`, `REJECTED`, or `ERROR` by downstream NiFi flow logic, any subsequent `POST /attachments/{parentTraceId}` returns *409 Conflict* with the message "Attachment window closed". Note that `PROCESSED` status does _not_ close the window — attachments are still accepted up to `attachments-max-count`.
