== Overview
This document outlines the requirements for the NiFi Extensions project, which provides JWT authentication processors and a REST API gateway for Apache NiFi. The project includes:

* **MultiIssuerJWTTokenAuthenticator** -- FlowFile-based JWT validation against multiple identity providers
* **RestApiGatewayProcessor** -- self-contained REST API gateway with embedded HTTP server, route-based dispatching, and RFC 9457 error responses
* **JwtIssuerConfigService** -- shared Controller Service for JWT issuer configuration and token validation

For detailed implementation specifications, see the xref:Specification.adoc[Specification Document].

== General Requirements

[#NIFI-AUTH-1]
=== NIFI-AUTH-1: REST API Support Enhancement

* Enable simplified REST API endpoint provisioning in NiFi
* Provide standardized JWT validation capabilities
* Support OAuth 2.0 token validation workflow

_Implementation details: xref:Specification.adoc[Main Specification]_

== MultiIssuerJWTTokenAuthenticator

[#NIFI-AUTH-2]
=== NIFI-AUTH-2: Core Functionality

* Name: MultiIssuerJWTTokenAuthenticator
* Extract JWT (JSON Web Token) from HTTP Bearer token header
* Validate token structure and signature
* Extract token information
* Route flow files based on validation results

_Implementation details: xref:specification/technical-components.adoc[Technical Components]_

[#NIFI-AUTH-3]
=== NIFI-AUTH-3: Token Validation Requirements

* Verify token structure (header, payload, signature)
* Validate token signature using issuer public key/certificate
* Check token expiration (exp claim)
* Verify token issuer (iss claim)
* Validate token audience if present (aud claim)
* Support multiple token issuers

_Implementation details: xref:specification/token-validation.adoc[Token Validation]_

[#NIFI-AUTH-4]
=== NIFI-AUTH-4: Multiple Issuer Support

* Configure multiple trusted issuers
* Store issuer-specific validation configurations
* Support different signing algorithms per issuer
* Allow dynamic issuer key/certificate updates

_Implementation details: xref:specification/token-validation.adoc[Token Validation]_

[#NIFI-AUTH-5]
=== NIFI-AUTH-5: Input Requirements

* Accept JWT token in flow file
* Support configurable token location in flow file
* Handle various token formats (Bearer, raw JWT)

_Implementation details: xref:specification/technical-components.adoc[Technical Components]_

[#NIFI-AUTH-5.1]
==== NIFI-AUTH-5.1: Token Location

* Extract token from Bearer Authorization header
* Format: "Bearer <token>"
* Support validation of header presence and format

[#NIFI-AUTH-7.2]
==== NIFI-AUTH-7.2: Token Validation Configuration

* Configure multiple issuers through UI
* Each issuer configuration consists of:
  ** Issuer Identifier (String)
  ** One of:
    *** JWKS endpoint URL for key retrieval
    *** Direct public key configuration for signature verification
* Support dynamic addition/removal of issuers
* Validate configuration inputs
* Provide clear error messages for invalid configurations

_Implementation details: xref:specification/configuration.adoc[Configuration]_

[#NIFI-AUTH-7.3]
==== NIFI-AUTH-7.3: Verification

* Provide a token testing interface in the UI
* Include a text area for pasting JWT tokens
* Add a "Verify Token" button to test token against current configuration
* For valid tokens:
  ** Display token validation success message
  ** Show decoded token payload (claims)
  ** Display which issuer validated the token
  ** Show what attributes would be added to the flow file
* For invalid tokens:
  ** Display validation failure reason
  ** Show detailed error information
  ** Suggest possible fixes based on error type
* Support testing without affecting processor state or flow files

_Implementation details: xref:specification/configuration-ui.adoc[UI Configuration]_

[#NIFI-AUTH-7.4]
==== NIFI-AUTH-7.4: Authorization Configuration

* Require Valid Token (Boolean)
  ** When true: Valid token results in success relationship
  ** When false: Token validation result is informational only
* Per-issuer authorization settings (configured per issuer, not globally):
  ** Required Scopes (List of String) — OAuth scopes that must be present in token
  ** Required Roles (List of String) — roles that must be present in token
  ** All configured requirements (scopes, roles) for the matched issuer must be met for success

_Implementation details: xref:specification/configuration.adoc[Configuration]_

[#NIFI-AUTH-7.5]
==== NIFI-AUTH-7.5: Configuration Properties

* List of Issuer Identifiers (Dynamic Property)

  ** Issuer Identifier
  ** JWKS endpoint URL or public key

_Implementation details: xref:specification/configuration.adoc[Configuration]_

[#NIFI-AUTH-7.6]
==== NIFI-AUTH-7.6: Static Configuration for Container Environments

* Support static configuration via properties files for container deployments
* Configuration format options:
  ** NiFi properties file (.properties)
  ** YAML configuration (.yml, .yaml)
* Static configuration must:
  ** Be active by default when present
  ** Override UI-based configuration when present
  ** Be displayed (read-only) in the configuration dialog
  ** Support all configuration options available in UI
  ** Include explicit documentation for container deployment scenarios
* Properties file location options:
  ** Standard NiFi configuration directory
  ** Environment variable pointing to configuration file
  ** JVM system property pointing to configuration file
* Configuration reload:
  ** Support periodic checking for configuration changes
  ** Allow runtime reload without restart
  ** Log configuration changes at appropriate level

_Implementation details: xref:specification/configuration.adoc[Configuration]_

== REST API Gateway Requirements

[#NIFI-GATEWAY-1]
=== NIFI-GATEWAY-1: Embedded HTTP Server

* Provide an embedded HTTP server (Jetty 12) within the NiFi processor
* Support configurable listening port and bind address
* Support optional SSL/TLS via NiFi's SSLContextProvider
* Accept HTTP requests directly without requiring HandleHttpRequest/HandleHttpResponse

_Implementation details: xref:specification/technical-components.adoc[Technical Components]_

[#NIFI-GATEWAY-2]
=== NIFI-GATEWAY-2: Multi-Route Configuration

* Support multiple named routes via dynamic properties using `restapi.<name>.<property>` pattern
* Each route defines: path (exact match), allowed HTTP methods, required roles, required scopes
* Default HTTP methods: GET, POST, PUT, DELETE
* Each route creates a corresponding NiFi relationship for FlowFile routing

_Implementation details: xref:specification/configuration.adoc[Configuration]_

[#NIFI-GATEWAY-3]
=== NIFI-GATEWAY-3: Per-Route JWT Authorization

* Authenticate all requests using Bearer JWT tokens via the shared JwtIssuerConfigService
* Support per-route required roles and scopes
* Return RFC 6750 compliant WWW-Authenticate headers on authentication failure
* Return 403 Forbidden for missing roles, 401 with `insufficient_scope` for missing scopes

_Implementation details: xref:specification/technical-components.adoc[Technical Components]_

[#NIFI-GATEWAY-4]
=== NIFI-GATEWAY-4: RFC 9457 Error Responses

* All error responses must use `application/problem+json` format per RFC 9457
* Each error type must link to human-readable documentation via the `type` field
* Support error types: 400, 401, 403, 404, 405, 413, 422, 500, 503

_Implementation details: xref:specification/error-handling.adoc[Error Handling]_

[#NIFI-GATEWAY-5]
=== NIFI-GATEWAY-5: CORS Support

* Support configurable CORS allowed origins
* Handle preflight OPTIONS requests automatically
* Add appropriate Access-Control-* headers to responses

_Implementation details: xref:specification/configuration.adoc[Configuration]_

[#NIFI-GATEWAY-6]
=== NIFI-GATEWAY-6: Management Endpoint

* Provide a `/metrics` endpoint on the gateway port
* Support Prometheus text format and JSON format (content negotiation via Accept header)
* Support optional API key authentication for the management endpoint
* Expose token validation, transport security, and gateway application metrics

_Implementation details: xref:specification/observability.adoc[Observability]_

[#NIFI-GATEWAY-7]
=== NIFI-GATEWAY-7: Request Size Limits

* Support configurable maximum request body size
* Return 413 Payload Too Large when exceeded
* Default limit: 1 MB

_Implementation details: xref:specification/configuration.adoc[Configuration]_

[#NIFI-GATEWAY-8]
=== NIFI-GATEWAY-8: Back-Pressure Integration

* Use a bounded request queue for NiFi flow integration
* Return 503 Service Unavailable when the queue is full
* Configurable queue size (default: 50)

_Implementation details: xref:specification/technical-components.adoc[Technical Components]_
